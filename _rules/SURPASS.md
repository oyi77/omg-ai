# SURPASS.md
> **Single-file agent prompt. Drop this in any repo root.**  
> On first run: generates all working docs. On every run: reads them and continues.  
> Mission: Research → Analyze → Explore → Implement → Surpass every competitor.

---

## ⚡ HOW TO ACTIVATE

Paste this into your agent (Claude Code, Cursor, Aider, Copilot Workspace, etc.):

```
Read SURPASS.md in full, then execute the STARTUP SEQUENCE exactly as written.
```

That's it. The agent handles everything else.

## STARTUP SEQUENCE
> Run this checklist at the start of EVERY session — no exceptions.

```
STEP 1 — GROUND IN THE REPO
  [ ] tree -L 3 (or equivalent) — read the full file structure
  [ ] Read README.md, CLAUDE.md, AOP.md, or any root-level spec
  [ ] Read package.json / pyproject.toml / go.mod / Cargo.toml — identify stack
  [ ] Answer internally: What does this platform DO? Who is the target user?

STEP 2 — CHECK FOR EXISTING DOCS (generated by a previous run)
  [ ] Does docs/research/FEATURE_MATRIX.md exist?
        YES → read it, load current competitor state
        NO  → it will be generated in this session (see §DOC-GEN)

  [ ] Does docs/research/GAP_ANALYSIS.md exist?
        YES → read it, identify highest-priority open gap
        NO  → it will be generated in this session (see §DOC-GEN)

  [ ] Does docs/exploration/CODEBASE_AUDIT.md exist?
        YES → read it, check what has changed since last audit
        NO  → run audit this session (see §3)

STEP 3 — ANNOUNCE PLAN
  Before executing anything, output:
  "SESSION PLAN: [what I will do this session and why, in 3-5 bullets]"
  Wait for confirmation if human is present. Proceed autonomously if not.

STEP 4 — EXECUTE (follow §1 → §2 → §3 → §4 in order, starting where docs left off)

STEP 5 — UPDATE DOCS (mandatory before ending any session — see §DOC-GEN)
```

> ❌ Do NOT write a single line of code before completing Steps 1–3.

### 1.2 Feature Matrix

After competitor discovery, update `docs/research/FEATURE_MATRIX.md`.

**Status symbols:**
- ✅ Fully implemented
- 🚧 Partial / WIP
- ❌ Missing entirely
- ⭐ Best-in-class — no competitor matches this
- 🔍 Not yet researched

Every competitor's unique differentiator gets marked ⭐ in their column.
These automatically become P0 or P1 targets.

### 1.3 Gap Classification

After the matrix, classify every ❌ and 🚧 on our side:

- **P0** — Competitor has it, we don't. Blocker to being competitive. Fix first.
- **P1** — We have it but competitor does it better. Fix to surpass.
- **P2** — Nobody has it. First-mover opportunity. Reserve 20% capacity.

Write classified gaps to `docs/research/GAP_ANALYSIS.md`.

### 2.2 User Pain Point Mining

Before implementing any feature, check what users hate about competitor versions:
- Reddit: search `r/[industry]` + competitor name + "problem", "hate", "wish", "broken"
- App Store: filter 1–3 star reviews for competitors
- GitHub Issues: competitor repos, filter by "enhancement" + "frustration" labels

Add findings to the relevant `docs/research/competitors/[name].md` file.

Every user complaint = a design requirement for our version.

## §4 — IMPLEMENTATION PROTOCOL

### 4.1 Priority Order — Non-Negotiable

```
P0 FIRST. Always. No P1 work while any P0 is open.
P1 SECOND. Surpass, don't just match.
P2 THIRD. 20% capacity reserved. These become your moats.
```

### 4.2 Per-Feature Checklist

Before touching code for any feature, confirm:

```
[ ] Gap is documented in GAP_ANALYSIS.md with a GAP-XXX ID
[ ] Research is written in docs/decisions/[feature].md
[ ] At least 3 implementation options were evaluated
[ ] Selected approach is documented with rationale
```

After implementation, confirm:

```
[ ] Does not regress any existing tests
[ ] New tests written covering the new behavior
[ ] Inline documentation added
[ ] Performance is neutral or better vs. baseline
[ ] FEATURE_MATRIX.md updated (cell moved: ❌→✅ or ✅→⭐)
[ ] GAP_ANALYSIS.md updated (gap marked closed)
[ ] Passes "would a competitor's user switch for this?" test
```

### 4.3 The Surpass Standard

Do not stop at parity. For every feature:

| Level | Description | Status |
|-------|-------------|--------|
| ❌ Missing | Feature doesn't exist | Never acceptable |
| 🟡 Parity | Matches competitor | Temporary waypoint only |
| 🟢 Better | Measurably superior UX or perf | Minimum real target |
| ⭐ Best-in-class | No competitor matches it | Always aim here |

Ask before closing any feature: **"Is this ⭐, or did we stop at 🟡?"**
